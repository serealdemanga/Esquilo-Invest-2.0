1. Visão geral

O que este manual cobre

Este guia mostra como deixar o AppScript pronto para responder ao app Flutter em produção local, sem alterar a regra de negócio e sem criar backend paralelo.

Fluxo real entregue
Papel do AppScriptServir HTML do dashboard atual e JSON para o app mobile no mesmo deploy.
O que será configuradoArquivos .gs, um arquivo .html, Script Properties, Web App e URL final /exec.
Resultado esperadoFlutter consumindo ?format=json&resource=... com o contrato já entregue.
Caminho mais seguroAtualizar o projeto AppScript existente do repositório, sem renomear arquivos.

Resumo funcional da integração

  • O Flutter lê a URL base do backend por APP_SCRIPT_BASE_URL.
  • O token opcional é lido por APP_SCRIPT_API_TOKEN.
  • O AppScript responde JSON quando recebe format=json.
  • Os recursos mobile entregues são dashboard, ai-analysis e health.
  • O app mobile não recria lógica de negócio; ele consome o mesmo getDashboardData().
Cenário recomendado: manter o projeto AppScript já existente e apenas aplicar os arquivos e ajustes listados neste manual. Recriar do zero é possível, mas exige copiar os arquivos de suporte também.

Resultado esperado ao final

  • A URL /exec?format=json&resource=health responde com ok: true.
  • A URL /exec?format=json&resource=dashboard devolve o payload usado na home mobile.
  • O Flutter deixa de mostrar o estado de configuração e passa a carregar patrimônio, score e radar.
  • A leitura da Esquilo IA fica disponível sob demanda quando as chaves da IA estiverem configuradas.

4. Passo a passo

Criação e cópia dos arquivos no AppScript

Use os nomes exatos abaixo. No editor do Apps Script, clique em +, escolha Script para arquivos .gs e HTML para Dashboard.html.

Parte 1
1

Abrir o projeto AppScript correto

Abra o projeto backend que já atende o dashboard web atual. O caminho mais seguro é atualizar esse projeto, não criar um projeto paralelo.

Objetivo: manter o mesmo backend para web + mobile.
2

Criar ou editar Config.gs

Crie um arquivo do tipo Script chamado Config ou substitua o conteúdo do arquivo existente.

Copiar de: apps_script/utils/Config.gs
3

Criar ou editar Backend_Core.gs

Esse arquivo precisa conter o doGet(e) que detecta format=json antes de renderizar o HTML.

Copiar de: apps_script/backend/Backend_Core.gs
4

Criar Mobile_Api.gs

Crie um arquivo Script chamado Mobile_Api e cole o conteúdo integral do arquivo do repositório.

Copiar de: apps_script/backend/Mobile_Api.gs
5

Garantir os arquivos de suporte do dashboard

Se você estiver recriando o projeto do zero, copie também os scripts que sustentam o payload da home mobile e do radar.

Copiar de: apps_script/services/Sheet_Readers.gs apps_script/services/Portfolio_Metrics.gs apps_script/services/Decision_Engine.gs apps_script/services/BigQueryService.gs
6

Garantir o bloco da Esquilo IA, se desejar

Para o recurso ai-analysis, copie os arquivos da IA e das chaves. Sem isso, a home mobile continua possível, mas a leitura sob demanda não fica operacional.

Copiar de: apps_script/services/AI_Service.gs apps_script/integrations/Api_Keys.gs
Não mude os nomes dos arquivos. O Apps Script agrupa tudo no mesmo namespace, então a forma mais segura de refletir a solução entregue é usar exatamente os nomes já existentes no repositório.
Criação: Script (.gs)Sem inventar novos nomes de arquivo
5. Passo a passo

Revisões, HTML e configurações obrigatórias

Esta etapa fecha o projeto AppScript para publicação real: HTML, propriedades, serviço avançado e revisão dos IDs.

Parte 2
7

Criar Dashboard.html se quiser preservar o web atual

Crie um arquivo do tipo HTML chamado Dashboard e cole o conteúdo do dashboard web existente.

Copiar de: frontend/html/Dashboard.html
8

Revisar APP_CONFIG_ em Config.gs

Confirme spreadsheetId, spreadsheetUrl e os nomes de abas em sheetNames. O mobile depende desse mesmo payload.

Campos mais críticos: spreadsheetId spreadsheetUrl sheetNames.actions sheetNames.funds sheetNames.previdencia sheetNames.preOrders sheetNames.aportes sheetNames.config sheetNames.dashboardVisual sheetNames.exportAuxiliar
9

Configurar Script Properties

Adicione as propriedades necessárias no projeto Apps Script. Use o painel equivalente de propriedades do script no editor atual.

MOBILE_APP_API_TOKEN = opcional, mas recomendado GEMINI_API_KEY = necessário para ai-analysis via Gemini OPENAI_API_KEY = opcional, fallback da IA OPENAI_MODEL = opcional EXTERNAL_MARKET_DATA_ENABLED = opcional
10

Habilitar BigQuery API se quiser manter a arquitetura atual

O arquivo BigQueryService.gs usa o serviço avançado BigQuery. Se ele não estiver habilitado, o backend tende a cair no fallback de planilha onde isso já está tratado.

Serviço extra: Apps Script Advanced Service: BigQuery API Também habilitar a BigQuery API no projeto Google Cloud associado, se necessário.
6. Mapeamento

De/para dos tipos de arquivo

Mapa rápido

Config.gs, Backend_Core.gs, Mobile_Api.gs

Criar como Script (.gs) no Apps Script e colar o conteúdo dos arquivos correspondentes do repositório.

Dashboard.html

Criar como HTML, com nome Dashboard, usando o conteúdo de frontend/html/Dashboard.html.

AI_Service.gs e Api_Keys.gs

Criar como Script (.gs) apenas se quiser manter o recurso ai-analysis ativo.

BigQuery API

Não é arquivo. É um serviço avançado do Apps Script que precisa ser habilitado para o caminho principal via BigQuery.

Bibliotecas

Nenhuma. A integração entregue não depende de biblioteca Apps Script externa.

Script Properties

Entram como configuração manual do projeto, não como arquivo separado.

Configurações necessárias depois de colar os arquivos

  • Revisar spreadsheetId e spreadsheetUrl em Config.gs.
  • Revisar os nomes de abas em APP_CONFIG_.sheetNames.
  • Definir MOBILE_APP_API_TOKEN se quiser proteger o acesso externo ao Web App.
  • Definir GEMINI_API_KEY para a aba de Insights com leitura remota.
  • Conferir se releaseName e versionNumber permanecem como Pocket Ops / 2.0.0.

Observação importante

O Flutter atual não exige que você edite um arquivo fixo com a URL do backend. Ele lê a URL no build/execução por --dart-define. Isso reduz retrabalho e evita hardcode.

7. Publicação

Como publicar como Web App

O app Flutter consome a URL final do deploy /exec. Sem essa URL publicada, o mobile não conecta.

Deploy
1

Abrir a publicação

No Apps Script, use o fluxo atual do editor: Deploy > New deployment e escolha Web app.

2

Executar como

Escolha executar como a conta proprietária do script. Assim o backend acessa a planilha e demais recursos com a mesma credencial do projeto.

3

Acesso para o Flutter

Para teste real no app Flutter, o deploy precisa ser acessível pelo dispositivo. Na prática, o caminho mais simples é acesso externo + MOBILE_APP_API_TOKEN.

4

Copiar a URL final

Depois de implantar, copie a URL do tipo .../exec. É essa URL que entra no Flutter. Não use a URL /dev para o app.

Exemplo: https://script.google.com/macros/s/SEU_DEPLOY/exec
Se a URL publicada não puder ser acessada pelo celular, o Flutter vai falhar mesmo que o código esteja correto. Na integração entregue, a URL precisa responder a requisições HTTP feitas pelo app fora do navegador logado.
8. Teste dos endpoints

Validação do backend online

Smoke test
EndpointExemplo de URLO que deve retornarComo saber se deu certo
health.../exec?format=json&resource=healthok: true, releaseName, versionNumber e updatedAtÉ o teste mais rápido para validar deploy, formato JSON e URL.
dashboard.../exec?format=json&resource=dashboardEnvelope com data.summary, score, categorySnapshots, updatedAtSe vier ok: true e o payload tiver a estrutura do dashboard, a home mobile já consegue abrir.
ai-analysis.../exec?format=json&resource=ai-analysisdata.analysis com texto da Esquilo IASe retornar erro de chave, o deploy está vivo, mas a IA ainda não está configurada.
Se você configurou MOBILE_APP_API_TOKEN, acrescente &token=SEU_TOKEN às URLs acima.
9. Conexão com o Flutter

Onde a URL entra no app mobile

mobile_app/

Arquivo que consome a configuração

O app lê a URL e o token em:

mobile_app/lib/core/config/app_environment.dart

As variáveis usadas são:

APP_SCRIPT_BASE_URL APP_SCRIPT_API_TOKEN

Como rodar com a URL publicada

flutter run --dart-define=APP_SCRIPT_BASE_URL=https://script.google.com/macros/s/SEU_DEPLOY/exec flutter run --dart-define=APP_SCRIPT_BASE_URL=https://script.google.com/macros/s/SEU_DEPLOY/exec --dart-define=APP_SCRIPT_API_TOKEN=seu_token
Validação no Flutter: a home deixa o estado de configuração, o cabeçalho mostra o backend ativo, e os blocos de patrimônio, score, alocação, radar e missão passam a renderizar.
Use a URL /execAPP_SCRIPT_BASE_URL via --dart-define

Problemas comuns

Abaixo estão os erros mais prováveis ao subir a integração atual, com a leitura direta do que revisar primeiro.

Operação
SintomaCausa provávelO que revisar
Erro Token mobile inválidoO token passado pelo Flutter ou pela URL não bate com MOBILE_APP_API_TOKEN.Revisar Script Properties e o valor usado em APP_SCRIPT_API_TOKEN.
Erro Recurso mobile não suportadoO parâmetro resource foi enviado com valor diferente de dashboard, ai-analysis ou health.Revisar a URL chamada ou o serviço Flutter.
HTTP 403 / acesso negadoDeploy sem acesso adequado para o celular ou URL errada.Revisar a publicação do Web App e confirmar a URL /exec.
JSON inválido ou HTML no lugar do JSONFaltou format=json na URL.Confirmar os parâmetros usados pelo Flutter ou pelo teste manual.
Planilha não localizadaspreadsheetId incorreto ou conta sem permissão.Revisar Config.gs e o acesso da conta proprietária do script.
Abas não encontradasNomes em sheetNames diferentes do que existe na planilha.Revisar cada chave dentro de APP_CONFIG_.sheetNames.
Erro da Esquilo IAChave Gemini/OpenAI ausente, inválida ou resposta fora do formato esperado.Revisar GEMINI_API_KEY, OPENAI_API_KEY, Api_Keys.gs e AI_Service.gs.

12. Conclusão

O que criar ou editar primeiro

Resumo prático

Obrigatório para subir o MVP

  • Config.gs
  • Backend_Core.gs
  • Mobile_Api.gs
  • Deploy Web App com URL /exec
  • APP_SCRIPT_BASE_URL no Flutter

Pode ficar para depois

  • Ativação da Esquilo IA, se você quiser testar apenas a home primeiro.
  • Rotinas fora do escopo mobile como CRUD completo, exportação e sincronização adicional.
  • Aprimoramentos de segurança e endurecimento operacional além do token mobile já previsto.
Ordem sugerida de execução: Config.gs → Backend_Core.gs → Mobile_Api.gs → propriedades → deploy → testes de health/dashboard → Flutter. Isso já coloca o MVP mobile em condição real de conexão.
