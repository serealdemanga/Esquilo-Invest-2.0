Visão Geral

Solução pensada para migrar com segurança

O app Flutter replica a leitura principal do produto atual em um fluxo mobile real, sem duplicar regra de negócio e sem substituir o backend existente. O AppScript continua centralizando os cálculos, leituras e payloads; o Flutter organiza navegação, renderização de cards táticos e consumo HTTP para Android.

Diagrama

Fluxo técnico da arquitetura

O mobile consome o AppScript por HTTP, interpreta envelopes JSON e transforma o payload em modelos Dart usados por telas e widgets independentes.

Flutter App
`PocketOpsApp`, `DashboardScreen`, tabs e detalhe por categoria.
Service Layer
`AppScriptDashboardService` monta a URL, chama o backend e normaliza erros.
AppScript Web App
`/exec?format=json&resource=...` responde dashboard, IA e health.
Dados e cálculo
Planilhas, métricas e motor analítico continuam centralizados no backend atual.

Decisões

Principais escolhas desta release

  • Aplicar a URL real do AppScript no app para simplificar o teste local.
  • Manter `--dart-define` como override sem desmontar a configuração existente.
  • Preservar o AppScript como backend principal, sem duplicação de lógica.
  • Isolar a home em cards táticos legíveis, otimizados para leitura vertical.
  • Tratar HTML de login do Google como erro explícito de integração, e não como falha genérica de parse.
Integração

Conexão Flutter + AppScript

A base URL ativa desta execução foi aplicada diretamente na configuração do app. O objetivo é garantir um APK debug pronto para validar a integração real no Android sem depender de placeholders.

Endpoint base aplicado https://script.google.com/macros/s/AKfycbya0mKpePyZV79S3ta0hJf0fhhwWfeUFwfopMOsZRSuqT7EF8LiTz6dV2suOuJNWve-/exec
Ponto Implementação atual Observação operacional
Base URL `AppEnvironment.appScriptBaseUrl` Usa a URL real como padrão e ainda aceita override por `--dart-define`.
Chamadas remotas `AppScriptDashboardService` Monta query com `format=json` e `resource=dashboard|health|ai-analysis`.
Tratamento de erro Exceções de integração normalizadas no app Quando o backend devolve HTML de login, a mensagem aponta o problema de publicação/acesso.
Status desta execução A URL foi aplicada corretamente no Flutter e o build Android segue íntegro. Na validação remota, porém, o endpoint publicado em /exec respondeu uma página HTML de login do Google em vez de JSON, indicando bloqueio de acesso do Web App para o app mobile nativo.
